# table-duo-vr 要件定義

Meta Quest 3 上で、**テーブルを囲む2人の非対称マルチプレイ VR**。
片方はフルアバター、片方は手だけのアバター。両者ハンドトラッキングで指・手首が動く。胴体位置は固定。

作成: 2026-06-09 / 改訂: 2026-06-10（設計監査反映・fixed-cam-vr リポジトリへ同居化）

## 0. 同居設計（fixed-cam-vr 内サブプロジェクト）

別 Unity プロジェクトを立てるコストを避け、fixed-cam-vr に同居する。干渉防止の規約：

- コードは **`Assets/TableDuo/`** に自己完結（Scenes / Scripts / Prefabs）。namespace は `TableDuoVr.<Feature>`
- asmdef は `TableDuoVr.*` で独立。**FixedCamVr.\* アセンブリとの相互参照は禁止**（references に入れない）
- シーンは `Assets/TableDuo/Scenes/TableDuoMain.unity`。`Main.unity` と Build Settings は排他運用（ビルドする側のシーンだけ登録）
- パッケージ追加は `com.unity.netcode.gameobjects` 1.12.0 のみ。Meta XR / URP 設定は共用
- Editor メニュー: `Tools/FixedCamVr/Setup/Setup TableDuo Scene`（シーン一式を冪等生成）

## 1. 確定コンセプト

| 項目 | 内容 |
|---|---|
| 環境 | 共有テーブルを向かい合わせで囲む |
| 人数 | 2人 / 同一 WiFi(LAN) マルチプレイ |
| プレイヤーA＝**人側**（人役/フル/`Role.Full`/Seat0/tdv_role full） | フルアバター（頭＋上半身＋両手、胴体位置固定）・**発話可** |
| プレイヤーB＝**手側**（手役/手だけ/`Role.Hand`/Seat1/tdv_role hand） | 手だけアバター（既定 右手・頭マーカー無し、定位置）・**発話不可** |
| 入力 | 両者ハンドトラッキング（指・手首の位置/回転） |
| 移動 | 胴体固定（着席・テレポート無し）。頭と手だけ動く |
| ゲーム性 | 対等（両者とも同一の操作能力。非対称は見た目のみ） |
| 表示モード | 完全 VR（パススルー MR 不採用） |
| 音声 | **生声（同室前提）。VoIP は実装しない**（v1） |

### 前提の明文化

- **2人は同室にいる想定**（同一 WiFi ＝ 同じ家/部屋）。会話は現実の空気経由
- 物理テーブルの共有はしない（Shared Spatial Anchor 不使用）。仮想の席は向かい合わせだが現実の立ち位置は互いに独立 → 現実での身体衝突は起きない
- B の「手だけ」にも**小さな頭マーカー**（浮遊する光点等）を付ける。視線・頷きの社会的キューが完全に消えると会話が一方通行になるため。pose は同期済みなので追加コストほぼゼロ

## 2. 技術スタック

| レイヤ | 採用 | 理由 |
|---|---|---|
| エンジン | Unity 2022.3.62f2 LTS | fixed-cam-vr と統一 |
| RP | URP | 同上 |
| XR | Meta XR All-in-One SDK | `OVRHand`/`OVRSkeleton` 公式・実機実績 |
| ネットワーク | Netcode for GameObjects **1.x 系に固定**（1.12+） + Unity Transport | NGO 2.x は Unity 6 向け。2022.3 LTS は 1.x 必須 |
| 接続 | 直接 IP over LAN（リレー/クラウド不使用） | 同 WiFi 前提なら直結が最も堅牢 |
| アバター | 自作シンプル | 非対称（手だけ/胴体固定）を素直に作れる。Meta Avatars SDK は過剰 |
| ターゲット | Android / IL2CPP / ARM64 | Quest 3 ネイティブ |

## 3. ネットワーク設計

### トポロジ / 接続

- ホスト兼プレイヤー1人 + クライアント1人（host-client、**ホスト権威**）
- 接続 UI は**手動 IP 入力から**。UDP ブロードキャスト discovery は後フェーズ
- Android `INTERNET` パーミッション必須

### 同期方式（重要・確定事項）

- **per-bone の NetworkTransform は禁止**。手 2 本 ×24 bone を個別 NetworkTransform にすると破綻する
- アバター状態は**単一の `INetworkSerializable` struct（AvatarState）**にまとめ、**NGO tick 30Hz** で送信、受信側は補間：

  ```
  AvatarState {
    headPos: Vector3, headRot: Quaternion
    wristPosL/R: Vector3, wristRotL/R: Quaternion
    boneRotL/R: half4[24]   // OVRSkeleton bone ローカル回転（half 精度）
    handTrackedL/R: bool    // トラッキングロスト通知
  }
  ```
  概算 ~400B × 30Hz ≈ 12KB/s/人 — LAN では誤差
- **量子化（smallest-three 等）はやらない**（v1）。LAN 帯域に対し過剰最適化。プロファイルで必要になった時だけ「後の最適化」として実施
- テーブル上オブジェクト: transform on-change 同期

### 掴みの裁定（確定事項）

- **grab はクライアントの「要求」**であり「主張」ではない。`GrabRequest` ServerRpc → ホストが裁定 → 勝者に ownership 付与
- 同時掴み（~100ms 以内）は**先着勝ち**。敗者の手はすり抜ける（ソフトフェイル、エラー表示なし）
- 解放も同様にホスト経由

### ライフサイクル（v1 ルール・確定事項）

- **切断・HMD 外し（近接センサスリープ）・アプリ中断 → セッション死亡。両者再起動**
- 再接続・ホストマイグレーションは実装しない（スコープクリープ防止のため明記）
- 開発時はホスト機の近接センサを `adb shell am broadcast -a com.oculus.vrpowermanager.prox_close_disable` 等で無効化してスリープ事故を回避

## 4. ハンドトラッキング

- `OVRHand` + `OVRSkeleton` で bone 取得 → AvatarState 送信 → 相手側で再構築
- ピンチ/グラブのジェスチャ検出（インタラクション判定）
- トラッキングロスト時: `handTracked=false` を同期し、リモート側は手をフェードアウト（点滅させない）
- **一人称の自分の手はローカルのトラッキングをそのまま描画**（ネット往復させない。最重要のレイテンシ決定）

## 5. アバター表現（段階導入）

**v1（両者共通）: 頭＋浮遊する両手。** 腕・胴体なし。

- A（フル）: 頭メッシュ＋両手 → 後続フェーズで胸/肩（席固定アンカー）を追加 → 腕はさらに後
- B（手だけ）: 両手＋最小の頭マーカー（小さな光点/リング）。胴体・腕は永久に描かない
- **腕の 2 ボーン IK は当面延期**。頭＋手首の 2 点入力では肘の pole-vector が推定頼みになり、破綻した肘は「腕なし」より presence を壊す。胴体が入って見た目が安定してから、見栄えする場合のみ追加
- 非対称は「B を描かない」という**省略で実装**される — 追加コストほぼゼロ、これがこの設計の利点
- シェーダ: アバターは matcap か簡易ランプ（1m 距離の Unlit は安っぽい）。小物・環境は URP/Unlit 基本

## 6. 空間 / 座席設計

- 完全仮想テーブル。各プレイヤーは自席にスポーン。物理部屋とは無関係
- **席＝固定リグ原点**。HMD 並進は頭にだけ反映、席は動かさない＝「胴体固定」を自然に満たす
- 起動時 recenter で正面をテーブルへ向ける
- **OS recenter（Oculus ボタン長押し）対応必須**: `OVRManager.display.RecenteredPose` を購読し、発火したら席アライメントを即再適用。放置するとセッション中にテーブルがズレる

## 7. インタラクション

- テーブル上オブジェクトをピンチ/グラブで掴む（キネマティック。物理は必要になったら）
- ownership はホスト裁定（§3）
- **ゲームルール設計上の制約**: 両手を重ねる・触れ合わせる姿勢を持続させるルールは避ける（Quest のハンドトラッキングはオクルージョン・接触で劣化する）。交互ターン制 or 自分のゾーン内操作を基本にする

## 8. 開発ループ（最重要・監査で最大の欠落と指摘された項目）

2台実機ビルド往復は 1 イテレーション 10 分超で開発が停滞する。**3 層の検証パス**を最初から用意する：

| 層 | 構成 | 検証できること | HMD 必要数 |
|---|---|---|---|
| L0 | Editor 2 プロセス（ParrelSync 相当 or ビルド済み PC クライアント）＋ **FakeHandDriver**（記録済み手カーブ再生） | NGO 同期ロジック・掴み裁定・struct 序列化 | 0 |
| L1 | Editor ホスト（**Quest Link**、Link 経由ハンドトラッキング）＋ Quest 実機クライアント | 実ハンドデータでの同期・見た目 | 1 |
| L2 | Quest 実機 ×2 | レイテンシ実感・熱・両者体験 | 2 |

- `FakeHandDriver`: 実機で記録した AvatarState 列を再生する MonoBehaviour。L0 の主役
- 日常イテレーションは L0/L1。L2 は各フェーズの締めだけ
- デバッグ可視化: Editor ホスト時はフラットなスペクテータカメラをミラー表示。実機は scrcpy

## 9. パフォーマンス目標

- 90 FPS 維持 / Update 内アロケーション 0（AvatarState はバッファ再利用）
- ドローコール 100 以下
- NGO tick 30Hz（60Hz はホスト Quest の熱・バッテリと相談。30+補間で足りる見込み）

## 10. 段階的実装計画

| Phase | 内容 | 完了条件 |
|---|---|---|
| 0 | プロジェクト疎通・ハンドトラッキング単体・**FakeHandDriver と記録機構**・Link 検証パス確立 | Editor(Link) と実機で自分の手が動く。手データの記録/再生ができる |
| 1 | NGO LAN 直結・頭 pose 同期 | **L0 で 2 プロセス同期 → L1 で Editor↔実機**。相手の頭が見える |
| 2 | AvatarState（手 root+指）同期・リモート手描画 | 相手の指が動いて見える（L1 で検証） |
| 3 | アバター v1（両者 頭+手、B は頭マーカー）＋テーブル＋掴み（ホスト裁定） | 2人で物を渡せる。役割が見た目で分かる |
| 4 | アバター差別化（A に胴体、腕は見栄え次第）・recenter 対応・ポリッシュ | A/B の非対称が明確。OS recenter してもズレない |
| 5 | 接続 UX（discovery）・perf パス（必要なら量子化）・ゲームルール実装 | 体験として成立 |

旧計画から: 旧 Phase 3(アバター)+4(掴み) を統合（掴みが同期の検証を早める）。量子化を Phase 2 から Phase 5 の条件付きへ降格。

## 11. やらないこと（DEFER・明示）

| 項目 | 理由 |
|---|---|
| 量子化/圧縮 | LAN 帯域に過剰。プロファイルで必要になったら |
| 腕の 2 ボーン IK | 肘推定の破綻リスク。胴体安定後に見栄え次第 |
| 物理ベース掴み | キネマティックで十分 |
| 再接続・ホストマイグレーション | v1 は「セッション死亡→再起動」 |
| VoIP | 同室前提・生声 |
| Shared Spatial Anchor / 物理テーブル重ね | 同室運用が固まったら検討 |
| UDP discovery | Phase 5。手動 IP で開発は回る |
