### 起動 runbook（PC から）

```powershell
# 1. ホスト役の Quest（シリアル A）— 例: 人役 (full)
adb -s <serialA> shell am start -n com.roiril.tableduo/com.unity3d.player.UnityPlayerActivity -e tdv_mode host -e tdv_role full

# 2. クライアント役の Quest（シリアル B）— 手役 (hand)
adb -s <serialB> shell am start -n com.roiril.tableduo/com.unity3d.player.UnityPlayerActivity -e tdv_mode client -e tdv_ip <hostQuestのIP> -e tdv_role hand

# 役割交代（HMD はそのまま、role を入れ替えて再起動）
adb -s <serialA> shell am force-stop com.roiril.tableduo && （上記を role を逆にして再実行）
# → tools/tableduo-role-swap.ps1 で自動化（Phase 5 参照）

# フェーズマーク（ホスト宛て）
curl "http://<hostQuestのIP>:7780/mark?label=<任意ラベル>"

# セッション後のログ回収
adb -s <serialA> pull /sdcard/Android/data/com.roiril.tableduo/files/ ./logs/<日付_ペアID>/
```

- 参加者ID / ペアID を刻む場合は `-e tdv_pid <ID> -e tdv_pair <ID>` を追加（CSV ヘッダとファイル名に入る）
- **`tdv_preplace` はデバッグ用（各席に静的アバター先置き）。本番セッションでは付けない（OFF）**

近接センサ無効化（装着外で動かす検証時のみ。実セッションでは不要）。

### 観戦（Spectator・任意）

ファシリテータは **PC の Unity Editor（Play）を 3人目の client（`tdv_role spectator`）として接続**し、両プレイヤーを固定俯瞰カメラでライブ観察できる（席を持たない seatless ロール・アバターは描画されない。実装 → `.claude/plans/2026-06-29_table-duo_spectator.md`）。

- Editor 側: TableDuoMain を開き ConnectionManager を autoMode=Client / studyRole=Spectator / IP=host Quest にして Play
- **⚠ 前提: Link/HMD 無しの Editor で OVR シーンを Play するとハングする**（既知）。観戦は L0 モード（`enableL0InEditor` or `tdv_l0=on`）か Standalone ビルドの CLI 起動で行う
- **⚠ 観戦 PC の `tdv_hand`（studyHandVariant）も両 Quest と同じ値にする**: 手の見た目は端末ローカル描画のため、観戦だけ違う値だと観戦記録の見た目が条件と食い違う。なお観戦 PC は手キャプチャが無いため Realistic/Robot は自動で Default 白手にフォールバックする（警告ログが出る）— **見た目条件の映像記録は scrcpy（人役 HMD ミラー）を正とする**
- 各フェーズの 👁 観察はこの観戦画面で行える（scrcpy ミラーと併用可）
- **⚠ 同意書に観戦・記録の説明が必要**: ファシリテータが観戦画面で観察し、スクリーンショット/リプレイを記録することを事前に開示・同意取得する（[consent-template.md](consent-template.md) に項目あり）。観戦の存在が行動を変えうる点は研究者判断で開示範囲を決める

## 1. セッション台本（40分目安）

> 表記: 🗣=ファシリテータの発話例 / 👁=観察ポイント / ⏱=目安時間 / 📌=フェーズマーク送信

### Phase 0: 同期クラップ ⏱1分

**装着してから接続（起動）する**のが基本（接続直後の相手アバターは休めポーズで出るが、手役本人のローカル手はトラッキング確立まで見えない — 装着済みなら数秒で揃う）。
両者装着・接続確認後、ファシリテータが **HMD マイクと室内レコーダーに聞こえる位置で1回大きく拍手**（映像にも入れる）。📌 `clap`
→ 全ストリーム（CSV / scrcpy 録画 / 室内音声）の時刻合わせ基準。

### Phase 1: 自由対面 ⏱2–3分 📌 `phase1`

🗣（人役へ）「向かいに『手』がいます。あいさつしてみてください」 — それ以上の指示はしない。
🗣（手役へ・事前カードで）「あなたは喋れません。手の動きだけで応えてください」

👁 人は最初どこを見て話すか / 手はどう「返事」を発明するか / 距離の取り方

### Phase 2: 物見せ課題 ⏱5分 📌 `phase2`

卓上は **Deep Sea Adventure（海底探検）ボードゲーム一式**（GLB モデル）: 潜水艦ボード＋宝物チップ・トークン・空気マーカー（静置）、**駒 ×2 とサイコロが掴める**。
🗣「駒（またはサイコロ）を取って、手に見せながら好きかどうか聞いてみてください」

👁 **物をどこへ向けて見せるか**（自動ログで取れるが、明らかな照準行動はシートにも記録）/ 手の Yes/No 表現

> 旧仕様（廃止・2026-06-30）: 絵カードデッキ（片面絵柄）＋食べ物プロップ。カード法線の照準ログ設計はこの旧仕様由来（掴んだ物の pose 記録は現行も同様）。

### Phase 3: 質問ゲーム ⏱5分 📌 `phase3`

🗣「手には『好きなもの』が1つあります（手役には事前カードで提示）。声で質問して当ててください。手はジェスチャーだけで答えます」

👁 質問が Yes/No 形式へ収束するか / 指数え・指差しの発生 / 正答までの質問数（**シートに記録**: 質問・応答ジェスチャ・成否）

### Phase 4: 海底探検 1 ラウンド ⏱8分 📌 `phase4`

卓上の海底探検を **1 ラウンド実際に遊ぶ**（2026-07-02 に旧「協調配置」から置換。全ピース掴める・
サイコロは離すと出目 1–3 がダイス上方に数字表示・出目は CSV `dice` 行に自動記録）。

🗣（両者へ）「ルールを簡単に説明します。交互にサイコロ 2 個を振って合計だけ駒を進め、チップを拾うか
選べます。拾うと帰りが遅くなります。空気（マーカー）が切れる前に戻ってください」
🗣（人役へ）「手はルールを知っていますが喋れません。分からなくなったら手に聞いてください」

- ルール裁定はアプリが強制しない（間違い→手役がジェスチャーで訂正する場面こそ観察対象）
- 空気マーカーの減算・手番管理は人役に任せる（手役は指差し・ジェスチャーで介入）
- ラウンドが崩壊してもデータ（👁 に記録して次フェーズへ）
- 盤面リセット: `curl "http://<hostIP>:7780/mark?label=reset_board"`（全ピースが初期配置へ戻る）

👁 手番・引き返し判断の交渉（「まだ行く？」「戻る？」）/ 数の伝達（チップ枚数・出目確認）/
指差し・OK/NG サインの発明 / ルール訂正のジェスチャー / 行き詰まりと打開

（余裕があれば難化: 空書禁止ラウンド）

### Phase 5: 役割交代 ⏱2分 📌 `swap`

HMD はそのまま、adb で role を入れ替えて再起動。`tools/tableduo-role-swap.ps1` で自動化（両機 force-stop → role を逆にして再起動）。新しい手役に禁止事項カードを渡す。

### Phase 6: 短縮再実施 ⏱10分 📌 `phase6-1` 等

Phase 1 → 3 → 4（海底探検は短縮 = 折り返しまで）を短縮版で。
👁 **手役経験者は人役になったとき、手への接し方が変わるか**（どこを見るか・話し方）

### Phase 7: 事後インタビュー（図指差し）⏱8分（個別に）

**図指差し（必須・RQ3 の主計測）**: 手の絵を渡し
🗣「手の『目』はどこにあったと思いますか。絵に丸をつけてください」

続けて両者に:

1. どこに向かって話して（見て）いましたか
2. （手役に）**あなたはどこから相手を見ているつもりでしたか**。自分の「目」はどこでしたか
3. 人アバター（箱頭）の目はどこにあると思いましたか
4. 伝わらなくて困った瞬間は？どう工夫しましたか
5. 手は「誰か」に感じましたか、「道具」に感じましたか
6. （手役に）喋れないことはどれくらいもどかしかったですか

### Phase 8: リプレイ見返し（stimulated recall）⏱25–30分 📌 議論を録音

PC モニタを3人で囲み、ReplayViewer で再生（手順は下記 runbook）。**当日中に実施**（記憶の鮮度が命）。

- イベント一覧（mark / grab / trackingLost）で頭出ししながら:
  - 手役へ 🗣「この動き、何を伝えようとしましたか」（意図）
  - 人役へ 🗣「これは何だと受け取りましたか」（解釈）
  - → **意図↔解釈 対応表**に記録（下記様式）
- **視点切替**: 「手の視点」プリセットで人役に「手から見たあなた」を見せ、反応を記録（RQ3 プローブ）
- 図指差し（Phase 7）はリプレイ**前**に済ませること（リプレイで「目の位置」の印象が上書きされるのを防ぐ）

意図↔解釈 対応表:

| 時刻 | 手の動き（要約） | 手役の意図 | 人役の解釈 | 一致? |
|---|---|---|---|---|
| | | | | |

最後に調査目的を説明（事後ブリーフィング）。**全体所要: 70–80分/ペア**。

### リプレイ運用 runbook

```powershell
# 1. セッション後、ホスト機からログ回収（CSV + tdv_replay_*.bin + 録画）
.\tools\pull-tdv-logs.ps1 -PairId pair01

# 2. Unity Editor: TableDuoMain を開く → [TableDuo]/ReplayViewer を有効化
#    Inspector: filePath = 回収した tdv_replay_*.bin / audioWavPath = 室内録音のwav（任意）
#    ConnectionManager.autoMode = None のまま → Play
# 3. 音声同期: イベント一覧の clap mark の時刻と、音声内のクラップ音の秒数の差を
#    audioOffsetSec に入力（±0.1/±1 ボタンで微調整）
```

## 2. 役割インストラクションカード（印刷用）

## 4. 手バリアントのブロック運用（決定 2026-07-02）

手の見た目 3 水準（Default / Realistic / Robot）は **within-pair の操作因子**。同じペアが 3 ブロックを体験する（設計 → [study-design.md](study-design.md) §2）。

- **各ブロックの起動**: 通常の起動 runbook に `-e tdv_hand <variant>` を追加（両機同じ値で起動）：

```powershell
# 例: Robot ブロック（人役 host / 手役 client とも同フラグ）
adb -s <serialA> shell am start -n com.roiril.tableduo/com.unity3d.player.UnityPlayerActivity -e tdv_mode host -e tdv_role full -e tdv_hand robot
adb -s <serialB> shell am start -n com.roiril.tableduo/com.unity3d.player.UnityPlayerActivity -e tdv_mode client -e tdv_ip <hostIP> -e tdv_role hand -e tdv_hand robot
# variant は default | realistic | robot
```

- **ブロック間**: 両機 force-stop → 休憩（HMD を外して数分）→ 次バリアントのフラグで再起動（役割交代と同じ運用。`tools/tableduo-role-swap.ps1` の起動に `-e tdv_hand <variant>` を足す）
- **順序割付**: ペア間でカウンターバランス（3 条件 = 6 順列。ペア数が 6 の倍数でなければラテン方格 3 順序）。**順序割付表は観察シートと同じ紙束に置き、ペアID とセットで管理**する
- **セッション中の切替は禁止**: 左 Y トグルは調査フラグ起動時に無効化済み（HandVariantWatcher）。万一切り替わったら CSV `handVariantChanged` イベントで検出し当該ブロックを除外
- **条件の記録**: CSV ヘッダ `# studyConfig: ... hand=<variant>` に自動で刻まれる（host=フル役端末の表示＝人役が見る手の見た目が操作対象）。観察シートにもブロック毎にバリアント名を記入

## 5. トラブル対応

| 症状 | 対処 |
|---|---|
| 接続が切れた | 両アプリ再起動（v1 仕様）。CSV はファイル分割されるので問題なし |
| 手のトラッキングが頻繁に落ちる | 照明を明るく / 袖をまくる / テーブル面の反射を避ける |
| 酔い・不調 | 即中断。データは破棄してよい |
| ガーディアン警告が出る | その場で立ち位置を調整（席アラインは自動追従） |
| 席がズレた（テーブルの位置がおかしい） | **両コントローラのグリップを同時に 3 秒長押し**で手動 recenter（頭を席の目線アンカーへ戻す） |
| 手が見えない | `HandPoseSampler.logDiagnostics=true` にして adb logcat の `[TDV-DIAG]` で tracked/confidence を確認（ハンドトラッキングは自分の手がカメラ FOV 内の時だけ高精度） |
